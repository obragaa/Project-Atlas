# 02 - Product Rules.md

# Product Constitution

> "Cada funcionalidade implementada no ATLAS deve respeitar estas regras. Elas possuem prioridade sobre preferências pessoais, tendências de design ou decisões momentâneas."

---

# 1. Princípio Fundamental

O ATLAS existe para aproximar o usuário de seus objetivos.

Toda funcionalidade deve gerar valor real.

Se um recurso não melhora a experiência do usuário, ele não deve existir.

---

# 2. Filosofia do Produto

O produto deve parecer inteligente.

Não apenas porque possui IA.

Mas porque todas as decisões da plataforma demonstram contexto.

O sistema deve antecipar necessidades sempre que possível.

---

# 3. Clareza acima da criatividade

Nunca sacrificar entendimento por estética.

Interfaces bonitas são importantes.

Interfaces compreensíveis são obrigatórias.

O usuário nunca deve pensar:

"O que esse botão faz?"

---

# 4. Uma funcionalidade = Um objetivo

Cada tela deve possuir um propósito principal.

Evitar telas que tentem resolver muitos problemas simultaneamente.

O foco aumenta a qualidade da experiência.

---

# 5. Zero funcionalidades vazias

Nenhuma tela poderá existir apenas para "encher o sistema".

Toda funcionalidade precisa entregar valor mensurável.

---

# 6. IA é um complemento

A Inteligência Artificial nunca substitui funcionalidades essenciais.

Ela potencializa a experiência.

O sistema deve continuar útil mesmo sem IA.

---

# 7. Personalização

Sempre que possível o produto deve utilizar contexto.

Exemplos:

* objetivo do usuário
* frequência de treino
* peso
* altura
* idade
* histórico
* progresso
* preferências

O usuário deve sentir que o sistema o conhece.

---

# 8. Menos cliques

Sempre procurar reduzir etapas.

A pergunta constante é:

"Existe uma forma mais simples?"

---

# 9. Performance é funcionalidade

Performance não é otimização.

Performance faz parte da experiência.

Qualquer ação deve transmitir velocidade.

---

# 10. Feedback imediato

Toda ação do usuário deve gerar resposta visual.

Exemplos:

* carregamentos
* salvamentos
* sucesso
* erro
* sincronização
* atualizações

O sistema nunca pode parecer travado.

---

# 11. Estados obrigatórios

Toda funcionalidade deverá possuir:

* estado inicial
* carregamento
* sucesso
* erro
* vazio
* atualização

Nenhuma tela poderá ignorar esses estados.

---

# 12. Acessibilidade

Todo componente deve ser acessível.

Requisitos mínimos:

* navegação por teclado
* contraste adequado
* foco visível
* leitores de tela
* tamanhos clicáveis

A acessibilidade não será adicionada depois.

Ela faz parte do desenvolvimento.

---

# 13. Responsividade

Não existe versão desktop.

Não existe versão mobile.

Existe apenas ATLAS.

A experiência deve funcionar naturalmente em qualquer dispositivo.

---

# 14. Consistência

Um componente criado hoje deverá funcionar exatamente igual em qualquer lugar do sistema.

Não criar variações desnecessárias.

---

# 15. Componentes reutilizáveis

Antes de criar um novo componente perguntar:

Ele realmente precisa existir?

Posso reutilizar outro?

---

# 16. Sem páginas mortas

Toda página deve incentivar uma próxima ação.

Exemplos:

Criar treino.

Conversar com o Atlas AI.

Atualizar progresso.

Concluir missão.

Nunca deixar o usuário sem direção.

---

# 17. Dados antes de decoração

Gráficos existem para informar.

Não para impressionar.

Todo gráfico deve responder uma pergunta.

---

# 18. Microinterações

Todas as interações importantes devem possuir feedback.

Mas nunca devem atrasar o usuário.

---

# 19. O Core

O Core é o centro do produto.

Toda informação importante deve estar acessível a partir dele.

O usuário deve conseguir entender sua situação atual em poucos segundos.

---

# 20. Atlas AI

O Atlas AI nunca deve responder apenas perguntas.

Ele deve orientar.

Explicar.

Incentivar.

Alertar.

Ensinar.

Personalizar.

Sempre considerando o contexto do usuário.

---

# 21. Missões

Missões não servem apenas para gamificação.

Servem para criar consistência.

As missões devem incentivar hábitos saudáveis.

Nunca dependência da plataforma.

---

# 22. Linguagem

A comunicação deve ser:

* clara
* objetiva
* respeitosa
* motivadora

Jamais utilizar linguagem agressiva.

Jamais culpabilizar o usuário.

---

# 23. Erros

Mensagens de erro devem explicar:

* o que aconteceu;
* o que o usuário pode fazer;
* quando tentar novamente.

Nunca mostrar mensagens técnicas ao usuário final.

---

# 24. Segurança

Toda informação do usuário deve ser tratada como sensível.

Mesmo quando não for obrigatória por lei.

---

# 25. Escalabilidade

Toda funcionalidade deve ser desenvolvida considerando crescimento futuro.

Evitar soluções temporárias.

Evitar código descartável.

---

# 26. Critério de aprovação

Nenhuma funcionalidade será considerada concluída enquanto não atender aos seguintes critérios:

* Resolve um problema real.
* Possui excelente UX.
* Está visualmente consistente.
* É acessível.
* Funciona em dispositivos móveis.
* Possui estados de loading, erro e vazio.
* Possui tratamento adequado de falhas.
* Está documentada.
* Está preparada para evolução futura.

---

# 27. Critério de excelência

Antes de finalizar qualquer entrega, responder honestamente:

* Eu teria orgulho de apresentar esta funcionalidade para um Staff Engineer?
* O design transmite qualidade premium?
* A experiência é intuitiva sem explicações?
* Existe alguma etapa desnecessária?
* Existe algum detalhe visual que possa ser refinado?
* A implementação respeita todos os documentos deste projeto?

Se qualquer resposta for negativa, a funcionalidade deve retornar para refinamento.

---

# 28. Regra absoluta

No ATLAS, qualidade sempre vence velocidade.

É preferível entregar uma funcionalidade extraordinária do que cinco funcionalidades medianas.

Essa regra deve orientar todas as decisões do projeto.
